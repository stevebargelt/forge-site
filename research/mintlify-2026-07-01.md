# Research: Mintlify as a documentation platform (revisit)

> **DECISION (2026-07-02): Do NOT adopt Mintlify for forge-site.** The build-vs-adopt verdict stands — forge-site stays on Astro + Starlight + the FW-9 CI-bot pipeline. Mintlify solves neither hard requirement (commander CLI reference, workflow JSON-Schema → docs), only replaces the `/docs` shell, and adds a hosted-platform dependency (GitHub App, Mintlify-hosted rendering, seat/credit pricing). Rationale below.

**Date:** 2026-07-01
**Status:** Research findings / decision input — reopens a narrow question, not the whole PRD
**Question:** Does Mintlify — currently working well on another project via `./mint.sh dev` local preview — change the 2026-06-05/06-25 verdict ("build bespoke, don't adopt a platform") for forge-site?
**Prior decisions on record:** `research/forge-docs-pipeline-2026-06-05.md` (bottom line: "Build it; don't adopt a platform") and `docs/prds/forge-website-platform.md` §"Platform choice" / §"Ruled out" (explicitly names Mintlify, Fern, Speakeasy, Stainless) and the epic `FW-9` non-goal "Adopting a third-party docs platform (Mintlify, Fern, etc.)."

---

## Bottom line

**Does not reopen the decision.** Mintlify is a good product for OpenAPI-first API docs with a polished authoring UX, and the `mint dev` local-preview loop is real and pleasant to use — that's plausibly why it "works well" on the other project. But it does not touch the two hard requirements that drove the original call, and it adds a platform dependency (GitHub App, hosted rendering, seat/credit pricing) our current Astro+Starlight+CI-bot design doesn't have. Nothing found here contradicts the 06-05/06-25 research; it sharpens it with primary-source detail.

## 1. What it is / architecture

- **Hosted-first product**, not a static-site generator you own end-to-end. The **CLI is named `mint`** (npm package `mint`, `npm i -g mint`), not `mintlify` — the older `mintlify` npm package is the deprecated CLI. [Installation](https://www.mintlify.com/docs/installation)
- **`mint dev`** runs a local preview server (default `http://localhost:3000`, `--port` to override) with live reload and local search. This is almost certainly what the other project's `./mint.sh dev` wraps — a thin shell script invoking `npx mint dev` (or the global `mint dev`) at the docs root. **No evidence of an official binary or script literally named `mint.sh`** in Mintlify's own docs/CLI/starter repo — it's a local convention, not a Mintlify-provided artifact. [Installation](https://www.mintlify.com/docs/installation) · [mint npm package](https://www.npmjs.com/package/mintlify) · [mintlify/starter](https://github.com/mintlify/starter)
- **Config file:** `docs.json` (successor to the deprecated `mint.json`, migrated via a CLI upgrade command). Unlike `mint.json`'s split fields (`navigation` + `tabs` + `anchors` + `versions`), `docs.json` is one recursive `navigation` tree — tabs/groups/pages are all nested under it. Large configs can be split into multiple files via `$ref`, resolved at build time relative to the referencing file. [Refactoring mint.json into docs.json](https://www.mintlify.com/blog/refactoring-mint-json-into-docs-json) · [Global settings](https://www.mintlify.com/docs/organize/settings)
- **Content → route mapping:** pages are `.mdx` files referenced by path (no extension) inside `docs.json`'s `navigation.groups[].pages[]`; the file path is the route. Same conceptual model as Starlight's file-based routing, but routes are declared centrally in `docs.json` rather than derived purely from the filesystem.
- **MDX authoring model:** standard MDX, React components usable directly in `.mdx` files (hooks included — `useState`, `useEffect`, etc.), reusable components can live in a `/snippets/` folder and be imported. **Nested imports are not supported** — every component used in a page must be imported directly into that page's MDX, no re-export chains. The renderer is `@mintlify/mdx`, an open-source thin wrapper over `next-mdx-remote` — this is genuinely open source and reusable outside Mintlify's platform, but it's the MDX *parser*, not the site/build/hosting stack. [React components](https://www.mintlify.com/docs/customize/react-components) · [mintlify/mdx](https://github.com/mintlify/mdx/) · [Launch Week II Day 3](https://www.mintlify.com/blog/launch-week-2-day-3)
- **Runtime is Next.js.** Mintlify's own hosting runs on Vercel/Next.js under the hood (confirmed by a joint Vercel/Mintlify case study). [Vercel case study](https://vercel.com/blog/mintlify-scaling-a-powerful-documentation-platform-with-vercel)

## 2. Content generation — the two hard requirements

**(a) Commander-based CLI reference generation: NO — not native, and no plugin surfaced for it.** Mintlify's generation story is exclusively OpenAPI/AsyncAPI-shaped: nothing in the docs, CLI reference, or scraper tooling ingests a commander command tree, a Cobra-style tree, or any non-HTTP CLI schema. Would need the exact same hand-rolled introspection generator (walking commander's `Help`/`Command` APIs) our own research already scoped in `research/forge-docs-pipeline-2026-06-05.md` — Mintlify buys nothing here.

**(b) Workflow JSON-Schema → docs: NO — not native, only OpenAPI-embedded schemas are supported.** Mintlify's schema-to-page feature (`openapi-schema` frontmatter field) generates a page **per schema inside an OpenAPI document's `components.schemas`** — it is not a standalone JSON-Schema-to-docs renderer. A bare JSON Schema file for our workflow YAML, with no enclosing OpenAPI document, is not a supported input. We would still need `@adobe/jsonschema2md` (or equivalent) exactly as already planned; Mintlify would only be the place the resulting MDX lands. [OpenAPI setup](https://www.mintlify.com/docs/api-playground/openapi-setup)

**OpenAPI support depth (for context, not our use case):** genuinely strong — `mint dev`/build reads an OpenAPI 3.0/3.1 doc referenced from `docs.json`'s `navigation` and renders interactive endpoint pages (params, response schemas, try-it playground, multi-language code samples) either dynamically at request time or pre-scraped into committed MDX via `npx @mintlify/scraping openapi-file`. Webhooks/callbacks supported on OpenAPI 3.1+. None of this reaches CLI or generic-schema docs. [OpenAPI setup](https://www.mintlify.com/docs/api-playground/openapi-setup) · [Autogenerating API documentation from OpenAPI](https://www.mintlify.com/blog/steps-to-autogenerate)

**Verdict on the two requirements: both still need the exact same bespoke generators the 06-05 research scoped.** Mintlify changes only where the *output* MDX lives, not who builds it.

## 3. Pipeline fit (FW-9 CI-bot-generates-MDX-then-PR)

- **The FW-9 shape is compatible in principle.** Mintlify's content model is just MDX files + `docs.json` in a git repo — nothing stops an external CI job from writing MDX/JSON into the repo and opening a PR via `peter-evans/create-pull-request`, same as the current Starlight plan. Mintlify doesn't require authoring through its web editor.
- **`_generatedFrom`-style frontmatter:** Mintlify's own docs only formally define `title`/`description` (and known optional keys like `icon`, `sidebarTitle`) in examples; there is **no documented allowlist/denylist for custom frontmatter keys**, and no evidence Mintlify rejects unrecognized keys. This is a reasonable "probably fine, unverified" — I could not find an authoritative source confirming custom keys are preserved and ignored vs. stripped by the build. Mark this **inconclusive** rather than confirmed. [Quickstart](https://mintlify.com/docs/quickstart.md)
- **Custom components:** supported (React-in-MDX, `/snippets/`), but with the nested-import restriction above — less flexible than Astro's island model, where a component can freely import other components/logic.
- **Drift/link checking:** the CLI ships a `mint broken-links` command (surfaced in CLI reference navigation, not deep-fetched here) — plausibly comparable to our planned `lychee` gate, but we have not verified its exact semantics (internal-only vs. external links, exit-code behavior for CI gating). Would need direct testing before relying on it as a CI gate.
- **What's genuinely missing vs. our design:** Mintlify has no concept matching our `content-only` vs `structural` auto-merge classifier (FW-19), render-convergence check (post-render gated re-run diff), or deterministic-emit `git diff --exit-code` drift guards (FW-3/FW-20). All of that is our own CI logic layered on top regardless of platform — Mintlify neither helps nor hinders it, since it operates purely on committed files after our pipeline has already produced them.

## 4. Hosting / deploy — Vercel compatibility vs. Mintlify's own hosting

- **Primary supported path is Mintlify-hosted.** Deployment is driven by installing Mintlify's **GitHub App** on the repo (needs org-owner/admin to install; the app requests **read/write permissions**, including the ability to push to protected branches unless branch-protection rules block it). Every push to the main branch auto-deploys; every PR gets an automatic preview deployment with a bot comment linking to it. [GitHub deploy](https://www.mintlify.com/docs/deploy/github) · [Preview deployments](https://learn.mintlify.com/courses/git-github/mintlify-previews)
- **"Vercel integration" is a reverse proxy, not a self-hosted build.** The documented Vercel pattern (`/docs/advanced/subpath/vercel`) is a `vercel.json` **rewrite** from `yoursite.com/docs` to `your-subdomain.mintlify.site/docs` — content is still served from Mintlify's infrastructure; your Vercel project is just a URL mask. This is materially different from "build the site and deploy the static output to our own Vercel project," which is how forge-site currently works. [Vercel subpath doc](https://www.mintlify.com/docs/advanced/subpath/vercel)
- **A true self-hosted/independent-deploy path exists only on Enterprise ("custom frontends")**: you own the frontend code/deployment, but Mintlify's blog post itself says content still funnels through **Mintlify's content engine** ("translates [MDX + docs.json] into Astro-compatible format") for snippet resolution, component extraction, and the AI/search features — i.e., even the most self-hosted option keeps Mintlify's backend in the loop, and it's gated behind Enterprise pricing/sales engagement. [Custom frontends on Mintlify](https://www.mintlify.com/blog/custom-frontends-on-mintlify)
- **Static-export-and-host-anywhere claims are unconfirmed by primary sources.** Some secondary sources (Medium walkthroughs) assert Mintlify "generates a static HTML site... hosted on Mintlify Cloud or Vercel, Netlify, GitHub Pages," but I found no official Mintlify doc describing a `mint build`/export command producing a portable static bundle, and an open (unanswered, as of fetch) GitHub discussion shows a user asking for the ability to serve raw static files under a Mintlify docs path — evidence the platform is not designed around "take the static output and host it yourself." Treat the free-tier deploy story as **Mintlify-hosted or reverse-proxied to Mintlify-hosted**, not "deploy anywhere like Astro." [GitHub discussion #1625](https://github.com/orgs/mintlify/discussions/1625)

**Net:** on Starter/Pro, forge-site would either move hosting to Mintlify (losing the existing Vercel deploy) or run a reverse proxy in front of Mintlify's hosting (adds a platform dependency without removing Vercel). Full independent hosting requires Enterprise and still isn't fully independent of Mintlify's backend.

## 5. Cost, lock-in, migration effort

- **Pricing (mintlify.com/pricing, fetched 2026-07-01):** Starter/Hobby is free (5,000 AI credits, custom domain, web editor, "Automations," MCP server) but has **no self-serve mid-tier shown on the pricing page anymore** — the next step up is Enterprise, "contact sales," with role-based permissions, SSO, SLA, and migration support. Third-party trackers (Ferndesk, Docsio, BunnyDesk) separately describe a **Pro plan around $250–300/month** (5 editor seats, 250 AI credits/mo, $20/seat beyond that, metered AI overages) sitting between the two — treat these as secondary/unverified since they don't match what the primary pricing page currently shows; get a live quote before budgeting. [Pricing](https://www.mintlify.com/pricing)
- **Lock-in surface:** `docs.json` schema, Mintlify's specific component set, its GitHub App/hosting model, and (for the AI/search features called out on the pricing page) its backend services are all Mintlify-proprietary. The MDX content itself is portable; the config, navigation model, and hosting integration are not.
- **Migration effort from Astro+Starlight:** nontrivial and one-directional-feeling — `docs.json` navigation would replace Starlight's frontmatter-driven auto-sidebar, the two authored-vs-generated MDX trees would need re-pointing, the plain-Astro marketing/landing surface (explicitly *not* Starlight, per the PRD, for design-control reasons) has no Mintlify equivalent at all — Mintlify only replaces the `/docs` shell, not a marketing site with GSAP/Remotion motion work. Adopting Mintlify would mean running **two systems** (Mintlify for `/docs`, Astro for landing/motion pages) rather than the current single Astro project, unless the marketing surface were also rebuilt to fit Mintlify's model (no evidence that's even supported — Mintlify has no analog to Astro islands/GSAP/Remotion integration).

## 6. Verdict input — does this reopen "build bespoke, don't adopt a platform"?

**No.** Specifically:

- Both hard requirements (commander CLI reference, workflow JSON-Schema docs) are **still unmet natively** — confirmed against Mintlify's current docs, not assumed. We would build/keep the exact same two generators (`Help`/`Command` introspection; `@adobe/jsonschema2md`) either way.
- The FW-9 CI-bot-PR delivery *shape* would still work content-wise, so this isn't a case of "the pipeline pattern is incompatible" — it's that **adopting Mintlify buys nothing on the part of the problem that was hard**, while costing:
  - a hosting-model change (Mintlify-hosted or reverse-proxy, vs. today's direct Vercel static deploy) or an Enterprise contract for a still-not-fully-independent self-host,
  - loss of the single-Astro-project design (marketing/motion surface has no Mintlify equivalent),
  - new recurring cost (free tier is thin; the realistic paid tier is Enterprise/contact-sales per the current pricing page),
  - a new proprietary config surface (`docs.json`, component set) to migrate onto and maintain drift-guards against.
- The one real positive signal — `mint dev` is a good local-authoring loop — is not differentiating; `astro dev`/Starlight's dev server already provides equivalent live-reload local preview today.

**Recommendation:** keep the 06-05/06-25 decision as-is. If "Mintlify is working well on another project" resurfaces, the actionable takeaway isn't "switch platforms," it's narrower: that project is very likely OpenAPI-first (Mintlify's actual strength), which forge-site is not — its hard content sources are a commander CLI and YAML workflow defs, not an HTTP API surface. Nothing here changes FW-9's design; no action items generated.
