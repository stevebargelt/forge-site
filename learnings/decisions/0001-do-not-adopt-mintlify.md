# 0001. Do not adopt Mintlify for forge-site docs

**Status:** Accepted
**Date:** 2026-07-02

## Context

forge-site's docs are generated by the FW-9 pipeline: forge CI emits versioned semantic JSON (commander CLI tree, workflow JSON Schema), forge-site renders that JSON into committed MDX served by Astro + Starlight, and a CI-bot PR flow (with a content-only/structural auto-merge classifier and render-convergence drift guard) delivers it. This design was chosen over adopting a third-party docs platform in `research/forge-docs-pipeline-2026-06-05.md`, and the exclusion was written into the PRD (`docs/prds/forge-website-platform.md`, §"Platform choice" / §"Ruled out") and encoded as an explicit non-goal on epic FW-9: "Adopting a third-party docs platform (Mintlify, Fern, etc.)."

That question was reopened narrowly on 2026-07-01 because Mintlify was reported to be "working well" on another project via a `./mint.sh dev` local-preview loop, prompting a fresh investigation (`research/mintlify-2026-07-01.md`) into whether that changed the calculus for forge-site specifically.

## Decision

forge-site stays on **Astro + Starlight + the FW-9 CI-bot generation pipeline**. Mintlify is not adopted.

Rationale:

1. **Mintlify solves neither hard requirement.** forge-site's hard content sources are a commander-based CLI tree and workflow JSON Schema. Mintlify's only native generation path is OpenAPI/AsyncAPI — it has no commander/Cobra-style CLI introspection and no bare-JSON-Schema-to-docs renderer (its schema pages only work embedded inside an OpenAPI document). We would still build the exact same bespoke generators (`Help`/`Command` walking, `@adobe/jsonschema2md`) that FW-9 already scopes; Mintlify would only change where the resulting MDX lands.
2. **It only replaces the `/docs` shell.** forge-site is one Astro project serving both the Starlight docs surface and a plain-Astro marketing/motion surface (GSAP interactive animation, Remotion video export) that has no Mintlify equivalent. Adopting Mintlify would mean running two systems instead of one.
3. **It adds a hosted-platform dependency we don't currently have.** The primary deploy path requires installing Mintlify's GitHub App (repo write access) and moves rendering onto Mintlify's infrastructure, with Starter/Pro seat- and credit-based pricing. The documented "Vercel integration" is a reverse proxy from our domain to a Mintlify-hosted subdomain, not a static export into our own Vercel project. A genuinely independent self-hosted path exists only on Enterprise, and even then Mintlify's content engine stays in the loop.
4. **FW-9's own drift guards are ours regardless of platform.** The content-vs-structural auto-merge classifier and render-convergence check are CI logic layered on top of committed files after generation — Mintlify neither provides nor replaces them.

## Consequences

- This ratifies FW-9's existing non-goal rather than changing scope; no backlog or pipeline design changes follow from this decision.
- The trigger for revisiting (Mintlify's good showing on another project) is attributed to that project's docs being OpenAPI-first, where Mintlify's `mint dev` authoring loop is a genuine UX strength — a docs-*shell* win, not a generation-*capability* win. It doesn't transfer to forge-site's CLI/workflow-schema content model, and `astro dev` already gives an equivalent local-preview loop for Starlight.
- If forge-site later gains hard content sources that are OpenAPI-shaped, this decision should be revisited against that specific surface rather than assumed to extend automatically.
